## UI Review — Home Page

### Issues found

**WARNING**

**1. `Hero.astro` — trust stats are hardcoded EN strings, not i18n**
`Hero.astro:53–70` — "AI agents", "DB drivers", "No Docker", "Local-first" are hardcoded English. Every other string on the page goes through `strings.ts`. This breaks the VI locale silently.

**2. `ViewAllFeaturesCta.astro` — copy is hardcoded, bypasses i18n**
`ViewAllFeaturesCta.astro:19–23` — title and description are inline ternary strings instead of `strings.featuresPage.*`. Inconsistent with every other component.

**3. `Architecture.astro` — protocol descriptions are hardcoded EN**
`Architecture.astro:19–23` — `protocols` array has hardcoded English descriptions ("CRUD, git, settings…", "PTY I/O…"). These won't translate. Minor since they're technical identifiers, but inconsistent with the i18n approach.

**4. `Architecture.astro` — metrics are hardcoded numbers**
`Architecture.astro:62–74` — "17.9k symbols", "300 exec flows", "30.6k relationships" are hardcoded. If these numbers change, they need to be updated in code rather than strings. Should be in `strings.ts` or at minimum extracted to a constant.

**5. `TrustRow.astro` — `box-shadow` on hover violates DNA**
`TrustRow.astro:68` — `.trust-card:hover` applies `box-shadow: 0 8px 24px color-mix(...)`. The DNA explicitly states no box-shadow as a depth signal. All other cards (glow-card, uc-card, layer) use only border-color + transform on hover.

**6. `UseCases.astro` — `.uc-card:hover` missing border-color change**
`UseCases.astro:76–78` — hover only does `translateY(-3px)` with no border-color feedback. Every other card in the system changes `border-color` on hover (glow-card pattern). The card feels unresponsive — the lift happens but there's no visual "activation" signal.

**7. `FAQ.astro` — answer indent doesn't align with question text**
`FAQ.astro:144` — `.faq-answer` has `padding: 0 24px 20px 64px`. The 64px left indent is meant to align with the question text (after the 24px number + 16px gap + some padding). But the number column is `width: 24px` and the gap is `16px`, so question text starts at `24px (summary padding) + 24px (num width) + 16px (gap) = 64px`. The math works, but on mobile the summary padding is the same 24px — so the answer aligns correctly. This is fine, but worth noting the magic number.

**8. `LicenseModel.astro` — double `border-block` creates doubled border with adjacent sections**
`LicenseModel.astro:42` — `.license-model` has `border-block: 1px solid var(--color-border)`. The section above (FAQ) has no bottom border, and the section below (GetStarted) also has no top border. This is fine structurally, but the `border-block` shorthand sets both top AND bottom. If the adjacent section also has a border, you get a doubled 2px line. Currently not doubled, but fragile — adding a border to an adjacent section will create a visual artifact.

**SUGGESTION**

**9. `Hero.astro` — "coming soon" chip sits inline with install label, competes visually**
`Hero.astro:36` — the warning chip is in the same flex row as the install label. On narrow screens this wraps awkwardly. The chip could move below the CopyBlock or be replaced with a tooltip/popover to reduce visual noise in the primary CTA area.

**10. `WhatIsSpecAde.astro` — comparison table `min-width: 480px` may clip on small phones**
`WhatIsSpecAde.astro:172` — the table has `min-width: 480px` inside a scroll container. On 360px-wide phones the scroll container itself is narrower than 480px, so the table scrolls horizontally. This is intentional (the `table-scroll` wrapper handles it), but there's no visual affordance (no scroll shadow or "swipe" hint) to tell users the table is scrollable.

**11. `Spotlight.astro` — `spotlight-copy` has `max-width: 38rem` but no equivalent constraint on visual slot**
`Spotlight.astro:67–71` — the copy column is capped at 38rem, but the visual slot has no max-width. On very wide screens (>1400px) the visual mock UIs can stretch beyond their designed proportions. The spotlight visuals (SpotlightClaw, SpotlightGoal, etc.) each manage their own width internally, so this is low risk but inconsistent.

**12. `Architecture.astro` — `layer:hover` uses `translateX(4px)` (horizontal slide)**
`Architecture.astro:127–130` — the horizontal slide is a deliberate design choice (layers "slide in" from left), but it's the only component on the page that uses horizontal translate on hover. All other cards use `translateY(-2px)` or `translateY(-3px)`. This is intentional differentiation, not a bug — but worth documenting in DNA.

**13. `GoalShowcase.astro` — section is very tall and dense**
The GoalShowcase section has a 3-tab scenario picker, milestone checklist, coder/watcher loop visual, and mock terminal all in one section. On mobile this stacks into a very long scroll. Consider whether the milestone checklist and loop visual need to be visible simultaneously, or if one could be progressive-disclosed.

**14. `MultiAgent.astro` — agent toggle buttons have no visible focus ring in the component's custom CSS**
The toggle buttons in MultiAgent use custom styling. Verify that the global `a:focus-visible` rule in `global.css` covers `button:focus-visible` too — if not, keyboard users navigating the agent toggles won't see a focus indicator.
